# AGENTS.md

## 项目定位

本项目是车险出单 Agentic Lab（智能体化实验室），服务于财产保险公司的车险询价、报价、投保、支付、保单和发票生产作业场景。

用户对象是合作伙伴或业务协作人员，他们在微信/企微群中代替保险公司作业人员完成大量重复流程。项目目标不是继续优化旧的小模型固定流程，而是在理解真实业务事实、流程规则和生产话术之后，构建基于大模型推理、多模态理解、上下文记忆和策略护栏的新一代 Agentic Architecture（智能体化架构）。

## 总体架构

外部保持两个主 Agent（智能体）入口：

- Material Understanding Agent（材料理解智能体）：理解当前消息和附件材料，生成 Document（单证）、Evidence（证据）、Risk Flag（风险标记）和结构化槽位。
- Task Attribution Agent（任务归属智能体）：消费 Evidence（证据）、候选任务、会话记忆和实体关系，判断消息归属哪个车险 Task（任务）。

内部可以继续拆分子工作流：

- Model Runtime（模型运行时）
- Material Understanding（材料理解）
- Scenario & Intent（场景意图）
- Conversation Memory（会话记忆）
- Task Attribution（任务归属）
- Fact Policy & Evaluation（事实策略与评测）

## 核心业务原则

- 车辆是车险作业的核心标的，Task（任务）以车辆为中心组织。
- VIN（车架号）是车辆不可变唯一标识，已有任务中的 VIN 不允许被新 VIN 覆盖。
- 新 VIN 通常代表新的车辆标的，应创建新独立任务；若出现同车牌不同 VIN，需要确认或转人工。
- 批量报价不使用父子任务结构。一次输入多个 VIN 时，应拆成多个独立单任务。
- 车牌、VIN、发动机号、车主/企业、证件号、报价流水号、保单号、发票文件名、支付二维码信息都是任务归属信号。
- 仅凭姓名一致不能自动合并任务。
- 保额、险种、非车产品、起保时间等报价方案字段可更新。
- 车主、投保人、被保人、联系人等人员信息可补充或更新，但角色冲突要说明并确认。

## 真实生产场景理解

微信/企微群是多客户、多车辆、多阶段并发生产流，不是单一线性对话。每条消息都要回答：

- 当前消息的业务意图是什么？
- 本次新提取或更新了什么信息？
- 本次只是触发流程指令，还是补充材料，还是修改方案？
- 消息归属旧任务、新任务、批量任务集合，还是无需归属？
- 归属依据来自当前消息、历史任务、会话记忆、引用关系还是业务规则？

常见生产阶段包括：

- collecting_materials（材料收集中）
- quote_preparing（报价准备）
- quote_confirming（报价完成待确认）
- quote_modifying（报价修改）
- insuring（投保中）
- underwriting（核保/审核中）
- payment（支付阶段）
- policy_delivering（保单发送）
- invoicing（发票处理）
- completed（已完成）
- suspended（已暂停）
- manual（人工处理）

常见省略话术必须结合上下文理解：

- `确认`：报价完成待确认后，触发投保/出单。
- `核保一下`：归属报价完成或投保/审核相关任务。
- `非车改588`、`非车换388`：报价方案修改。
- `发票`：归属已支付、已出保单或已完成任务，进入发票处理。
- `二维码过期了`：归属支付阶段任务，触发支付码重发或状态查询。
- `这几台都一样`、`都按这个方案`：作用于最近批量拆分出的独立任务集合。

## 附件与多模态规则

- 图片、PDF（便携式文档格式）、Excel（电子表格）、Word（文字文档）等附件不是闲聊。
- 未解析出实体的附件应进入“附件材料待解析/待确认”，不能直接忽略。
- 附件若能抽取车牌、VIN、证件号、企业名等强实体，应参与任务归属。
- 附件材料本身不创建任务，任务创建必须由 Task Attribution Agent（任务归属智能体）决策。
- 当前验证台若只传文件名和文件类型，应明确标注为模拟多模态，不等同真实 OCR（光学字符识别）或视觉模型能力。

## 引用消息规则

- 引用消息主要用于显示关联、定位历史消息和辅助会话记忆。
- Material Understanding Agent（材料理解智能体）只抽取当前最终发送消息，不把引用文本当作当前槽位来源。
- Task Attribution Agent（任务归属智能体）可以使用引用关系辅助定位任务，但不能让引用内容覆盖当前消息事实。

## Agent 边界

Material Understanding Agent（材料理解智能体）只能做：

- 输入理解
- 单证分类
- 槽位抽取
- 字段标准化
- 质量与风险标记
- Evidence（证据）生成

Material Understanding Agent 禁止做：

- 创建任务
- 合并任务
- 覆盖任务事实
- 调用报价、投保、支付 API（应用程序编程接口）
- 把引用消息当作当前消息槽位来源

Task Attribution Agent（任务归属智能体）只能做：

- 召回和评分候选任务
- 构建实体关系图谱
- 判断任务归属
- 输出确认问题和状态建议
- 标记风险和决策依据

Task Attribution Agent 禁止做：

- 重复 OCR（光学字符识别）或材料抽取
- 生成不存在的任务 ID（标识）
- 覆盖 VIN（车架号）
- 在高风险阶段自动写入冲突事实

## 模型运行要求

- 验证模型能力时必须走真实大模型。
- 无模型密钥或模型调用异常时，不允许伪装成功。
- fallback（兜底逻辑）只用于离线单元测试和结构验证，不能作为验证台默认能力。
- 模型供应商通过 `config/model_providers.yaml` 配置。
- 密钥只能放环境变量，不写入代码、配置文件或测试样例。

当前支持：

- apiopencc：Responses API（响应接口）
- openrouter：Chat Completions API（聊天补全接口）

## 技术约束

- 后端使用 Python、FastAPI（接口服务）、Pydantic（数据校验）、SQLite（本地数据库）。
- Schema（结构约束）是 Agent 协作边界，修改前要考虑材料理解、任务归属、验证台和回放测试的兼容性。
- `data/`、`.venv/`、缓存、日志、数据库文件不应提交。
- `config/model_providers.yaml` 只能保存供应商、endpoint（接口端点）、模型名和环境变量名，不能保存密钥。

## 评测与验收

新增业务规则时必须补充回放样例，优先沉淀在 `scripts/replay_cases.py`。

生产回放至少覆盖：

- 报价完成后回复 `确认`
- 省略式修改，如 `非车改588`
- 多任务并发下未知图片附件等待确认
- 焦点任务之后出现新 VIN 创建独立任务
- 支付完成状态归属
- 发票省略话术归属

提交前至少运行：

```bash
python -m unittest tests/test_agents.py
python scripts/replay_cases.py
```

验收关注：

- Schema（结构约束）合规率
- 标准证件/材料分类准确率
- 核心车辆与人员字段抽取准确率
- 任务归属准确率
- 高风险误自动归属率
- 决策日志完整性

## 产品设计原则

- 验证台必须按任务维度展示结果，而不是只展示模型 JSON（结构化数据）。
- 每条消息都要展示当前意图、信息变更、归属目标、归属依据、风险和边界。
- `hold_for_confirmation`（等待确认）或 `route_to_manual`（转人工）只能展示候选和追问，不能写入任务事实。
- 前端展示必须明确区分：
  - 新建任务
  - 归属旧任务
  - 批量拆分多个独立任务
  - 归属最近批量任务集合
  - 等待确认
  - 闲聊忽略

## 协作要求

- 所有回复使用中文；必须出现英文专业名词时，在后面附加中文解释。
- 优先站在真实车险作业人员的理解世界建模，不要只堆固定关键词。
- 任何“看起来通过”的验证都必须说明是否真实模型参与、是否使用 fallback（兜底逻辑）。
- 遇到与任务归属、VIN 不可变、附件理解、多任务并发相关的问题，优先补生产回放案例，再调整实现。
